Learn Gitflow

2019年4月3日
| Git
| 阅读

gitflow是一种规范，适合大项目开发

Github上开源了一个遵循这样规范的一个Git工具gitflow

初始化一个git

git flow init

然后会提醒一系列创建操作，类似

Which branch should be used for bringing forth production releases?
Branch name for production releases: [master]

Which branch should be used for integration of the "next release"?
Branch name for "next release" development: [develop]

How to name your supporting branch prefixes?
Feature branches? [feature/]
Bugfix branches? [bugfix/]
Release branches? [release/]
Hotfix branches? [hotfix/]
Support branches? [support/]
Version tag prefix? []
Hooks and filters directory? [/home/shelljo/go/src/github.com/WuShaoQiang/mega/.git/hooks]

我一般就是按照默认

到这里我们就有了两个分支master和develop

功能分支

如果要对软件实现一个功能，我们就可以创建一个feature分支

git flow feature start resetPassword

会出现

Switched to a new branch 'feature/resetPassword'

Summary of actions:
- A new branch 'feature/resetPassword' was created, based on 'develop'
- You are now on branch 'feature/resetPassword'

Now, start committing on your feature. When done, use:

     git flow feature finish resetPassword

这里提示，你已经进入到了一个新的分支叫feature/resetPassword，是基于develop分支的，这个时候我们就可以在这个分支上做功能的实现了

一般来说，功能分支都是自己本地管理，不用提交到远程上的，但是万一功能还未实现却需要做别的事情，我们也可以提交上去作为一个备份

git add -A
git commit -m "message"
git push origin

如果流没有对应的话，它会提醒你设置的--set-upstream

我们假设功能已经完成，现在需要提交到develop分支上，输入

git add -A
git commit -m "message"
git flow feature finish

会出现

Switched to branch 'develop'
Your branch is up to date with 'origin/develop'.
Updating be78ba5..99525b9
Fast-forward
 cmd/db_init/main.go | 2 ++
 1 file changed, 2 insertions(+)
To github.com:WuShaoQiang/mega
 - [deleted]         feature/test
Deleted branch feature/test (was 99525b9).

Summary of actions:
- The feature branch 'feature/test' was merged into 'develop'
- Feature branch 'feature/test' has been locally deleted; it has been remotely deleted from 'origin'
- You are now on branch 'develop'

这里可以看到gitflow自动把已经合并的分支给删除了，并且自动切换到了develop分支

这个时候我们可以将本地的develop分支推到远程的develop分支

git push origin

发布版本

发布版本是要上线到master分支的时候

git flow release start <version>

会出现

Switched to a new branch 'release/1.0.1'

Summary of actions:
- A new branch 'release/1.0.1' was created, based on 'develop'
- You are now on branch 'release/1.0.1'

Follow-up actions:
- Bump the version number now!
- Start committing last-minute fixes in preparing your release
- When done, run:

     git flow release finish '1.0.1'

可以看到也是自动创建和切换分支

当测试完毕的时候

git flow release finish

会出现

Switched to branch 'master'
Merge made by the 'recursive' strategy.
 cmd/db_init/main.go | 2 ++
 1 file changed, 2 insertions(+)
Already on 'master'
Switched to branch 'develop'
Your branch is up to date with 'origin/develop'.
Already up to date!
Merge made by the 'recursive' strategy.
Deleted branch release/1.0.1 (was 99525b9).

Summary of actions:
- Release branch 'release/1.0.1' has been merged into 'master'
- The release was tagged '1.0.1'
- Release tag '1.0.1' has been back-merged into 'develop'
- Release branch 'release/1.0.1' has been locally deleted
- You are now on branch 'develop'

我们还要执行

git push origin --all --follow-tags

一定要跟--follow-tags，不然无法创建

小结

gitflow当然不止这些，还有bug分支等，等用到了再总结。

关于.gitignore

笔者近期发现自己竟然把配置文件都给推到Github上了，真是惊了，于是乎想起有一个叫做.gitignore的文件，能够让我们在push到远程的时候不会把一些我们不想推的文件推上去，下面是关于.gitignore的配置，只是作为一个记录方便查阅，并不需要记

#               表示此为注释,将被Git忽略
*.a             表示忽略所有 .a 结尾的文件
!lib.a          表示但lib.a除外
/TODO           表示仅仅忽略项目根目录下的 TODO 文件，不包括 subdir/TODO
build/          表示忽略 build/目录下的所有文件，过滤整个build文件夹；
doc/*.txt       表示会忽略doc/notes.txt但不包括 doc/server/arch.txt
 
bin/:           表示忽略当前路径下的bin文件夹，该文件夹下的所有内容都会被忽略，不忽略 bin 文件
/bin:           表示忽略根目录下的bin文件
/*.c:           表示忽略cat.c，不忽略 build/cat.c
debug/*.obj:    表示忽略debug/io.obj，不忽略 debug/common/io.obj和tools/debug/io.obj
**/foo:         表示忽略/foo,a/foo,a/b/foo等
a/**/b:         表示忽略a/b, a/x/b,a/x/y/b等
!/bin/run.sh    表示不忽略bin目录下的run.sh文件
*.log:          表示忽略所有 .log 文件
config.php:     表示忽略当前路径的 config.php 文件
 
/mtk/           表示过滤整个文件夹
*.zip           表示过滤所有.zip文件
/mtk/do.c       表示过滤某个具体文件

整理来源于散尽浮华

前面我说了，我已经不小心将配置文件推到了Github上，所以我急忙补了一个.gitignore，但发现文件还是照样上传上去了，为什么呢？

因为.gitignore文件只对还没加入版本管理的文件起作用，所以我们要在git上删除，并且更新到远程，然后下次才能不将这个文件推上去

在Git上遇到了坑

笔者在某一天早晨发现，之前写的代码全没了。。 看了之前调用的命令才发现，不知道什么时候调用了git checkout把工作区所有的修改都还原了，也就是退回到和本地版本库里的版本。。。所以大家慎用git checkout
