Gopher 2019 Stack and Heap

2019年8月10日
| Golang
| 阅读

最近在多多回顾之前的Gopher大会视频，下面分享的主题是在2019 GopherCon Sg上由Jacob Walker带来的Understanding Allocations: The Stack and the Heap

这篇blog post只是为了总结

Stack

在Go里，每一个goroutine都有一个自己的stack，因此我们如何在goroutine之间传达stack的数据？

当然是用channel(推荐)或者全局变量(不推荐，且需要加锁)

但是这篇post主题不是这个，所以不加深讨论了

可能会有人有疑问，我只有一个main函数方法，没有起goroutine怎么办。可以这么理解，其实main函数本身就是一个goroutine

Heap

如果变量不存储在stack上，那么就是存储在了heap上

heap原则上不属于任何一个goroutine拥有，大家都可以对这里的变量进行引用

存储在heap上的变量会受到GC的管理(也就是垃圾回收)

Attention

其实在编写程序，我们更注重的是正确性，清晰性，可读性等

唯独我们真正需要Improve performance的时候，才需要考虑性能问题

什么时候要提升性能呢

  • 用benchmark证明程序确确实实没有达到需求的速度

Optimize for correctness first, not performance

How do we know where the variable is stored?

我们不知道，只有编译器知道，所以我们需要问编译器

go build -gcflags "-m"

但是我们可以根据一些通行的规则大概判断

Detail

我再简单的说说Go里面的stack

Go stack会分成两部分，一部分是valid，另一部分是invalid

从字面上也很容易看出，valid的内存才是真正有用的，可以被调用的.

我们不应该触碰到invalid内存里的东西，因为那是不安全的

什么时候是valid的呢

func main() {
    a := 1
    b := 1
    c := sum(a,b)
}

func sum(x,y) int {
    return x+y
}

一个很简单的程序

我们的main函数先声明了两个变量，这两个变量是在main函数的stack里存储

后面我们将这两个变量传入sum函数，这个时候stack会再增加两个变量(因为赋值操作)

所以现在我们有四个valid的变量

接着函数返回了，将值赋值给c，这个时候分为两部分

a,b,c都是stack中valid的部分

x,y都是stack中invalid的部分

我们在调用一个函数的时候，会产生stack

当我们函数返回的时候，stack会变成invalid

因此如果我们在外部还要继续引用函数所返回的值，那么我们是不能把这个放在invalid的stack中的

这个时候这个variable就escapes to the heap，到了heap上就可以被访问而不会出问题了

Points to remember

Go only puts function variable on the stack if it can prove a variable is not used after the function returns

Sharing down typically stays on the stack

Sharing up typically escapes to the heap
