Design Pattern Basic

2019年8月4日
| 设计模式
| 阅读

简单的记录一下最近遇到的设计模式，希望能够多点吸取一些软件设计上的经验

简单工厂模式

简单工厂模式和工厂模式还不一样，简单工厂模式和工厂模式的很主要的区别在于是否违反了开放-封闭原则。

在简单工厂里，Fac是一个类，它按照条件来产生不同的实体工厂，比如很简单的运算

type Operator interface {
    GetResult(num1 int, num2 int) int
}

type OperatorFac struct {

}

func (fac OperatorFac) CreateOperator(op string) Operator {
    switch op {
        case "+":
            return AddOperate{}
        case "-":
            return SubOperate{}
    }
}

type AddOperate struct {

}

func (addOp AddOperate) GetResult(num1 int, num2 int) int {
    return num1 + num2
}

type SubOperate struct {

}

func (subOp AddOperate) GetResult(num1 int, num2 int) int {
    return num1 - num2
}

func main() {
    fac := OperatorFac{}
    op := fac.CreateOperator("+")
    op.GetResult(1,2)   //3
}

当然，这些结构体是多余的，我这里只是示例一下

现在我能够输入我想要的op类型拿到对应的实例，对吧，这些实例都是符合一个接口的

如果我想加一个乘法进去，那么我们就要修改这个工厂，在它的判断条件里面加一个乘法选项，这就违背了开放-封闭原则，因为我们在扩展的时候修改了以前的代码

工厂模式

工厂模式弥补了简单工厂的缺陷，它不会违背开放-封闭原则

工厂模式：定义一个用于创建对象的接口，让子类决定实例化哪一个类。工厂方法使一个类的实例化延迟到了子类

我们还是以运算操作来示例一下

type Operator interface {
    GetResult(num1 int, num2 int) int
}

type OperatorFac interface {
    CreateOperator() Operator
}

type AddOperate struct {

}

type AddOperatorFac struct {

}

func (addOp AddOperate) GetResult(num1 int, num2 int) int {
    return num1 + num2
}

func (fac AddOperatorFac) CreateOperator() Operator {
    return AddOperator{}
}

func main() {
    fac := AddOperatorFac{}
    op := fac.CreateOperator()
    op.GetResult(1,2)   //3
}

这里我只实现了加法的功能，如果我要增加减法的呢？

只需要增加

type SubOperate struct {

}

type SubOperatorFac struct {

}

func (subOp SubOperate) GetResult(num1 int, num2 int) int {
    return num1 - num2
}

func (fac SubOperatorFac) CreateOperator() Operator {
    return SubOperator{}
}

这里我们可以看到，我们增加的时候并不需要去修改之前的代码

其实可能这两个例子来说，前面直接switch更加方便，但项目到达一定的规模时，或者开发者并不是同一个人时，第二种做法是有很大好处的，因为新来的人员不需要去改以前人写的代码，只需要实现Operator和OperatorFac这两个接口即可

你应该对比一下两种做法的main，第一种做法，我们是在函数里面判断到底要给哪种Operator，第二种做法是让main来决定到底要哪种Operator

其实两个模式都有优缺点，按需匹配吧

观察者模式 && 委托模式

观察者模式又叫做发布-订阅模式

观察者模式定义了一种一对多的依赖关系，让多个观察者对象同时监听某一个主题对象。这个主题对象在状态发生变化时，会通知所有观察者对象，使它们能够自动更新自己

我大概想了一下，在Go语言里面，这个有点像往订阅者的channel发送数据，当然也可以有其它的方法，比如调用订阅者的update函数

一般来说观察者有这几种方法

type Publisher interface {
    AddSubscriber(s Subscriber)
    RemoveSubscriber(s Subscriber)
    Notify()
}

而订阅者可以是这样

type Subscriber interface {
    Update()
}

这里的Update并没有参数，如果需要Publisher给Subscriber发送数据的话，推荐使用goroutine来起一个协程监听channel

委托模式

在观察者模式中，发布者和订阅者存在耦合，虽然我们可以用接口来定义两者，但它们两个接口还是耦合起来了

委托模式就是让这中耦合解除

想象一下，发布者不是需要让订阅者更新状态吗，我们直接传入更新状态的函数便可，这个函数我们可以抽象成一个接口类型

type UpdateFunc func()

订阅者的Update就跟着这个接口类型实现，然后让发布者调用传入的函数就可以了

func (p Publisher) Notify() {
    for _,f := range p.list {
        f()
    }
}

委托模式所搭载的方法需要是相同的一个格式
