Git用法总结

2019年3月15日
| Git
| 阅读

这篇博客是笔者总结学习git的过程，不是一篇专业性很强的文章，主要是笔记

主要参考廖雪峰的博客学习

笔者之前对Git有错误的理解(因为之前没学过，也只是听过)，总以为Github就是Git，其实Github是Git出来后才被创建的，为开源项目免费提供Git存储。

无Github,纯属本地的Git

— 那么什么是Git呢？

Git是目前世界上最先进的分布式版本控制系统

分布式：Git和其它集中式版本控制系统最大的区别，Git也可以有一个24小时开机的服务器来充当中央服务器，但这个服务器仅仅是为了让大家方便交换各自修改的。

版本控制：因为我们的代码要不断的完善，所以就有版本之分啦。

就算是我们自己在电脑上开发，也可以为自己代码创建版本库，用来记录修改和方便管理。

— 创建版本库

在想要创建版本库的文件夹里面，输入命令

git init

就可以看到文件夹里面多了一个隐藏的文件夹.git

添加文件到我们新建的版本库里面用

//将某一个文件添加进缓冲区
git add <file>  

//将.git文件夹 所在的文件 里所有的文件添加进缓冲区
//所以不管你是在某一个子文件夹用这个命令，执行效果是一样的
git add -A      

//这个就是将缓冲区的提交到版本库里，并且携带提交信息
//(强烈推荐要写message，这样可以提醒那个时候修改了什么)
git commit -m <message>

我们可以把代码所在区域分成

  • 工作区
  • 缓冲区
  • 当前分支

理解这几个工作区域会有助于理解命令的作用。

工作区：我们自己电脑编辑代码的地方

缓冲区：就是我们把修改过的文件放到一个区域装着，如果后面我又对这个文件做修改了，就可以继续往里面添加(同文件覆盖)

当前分支：就是我们使用commit后将缓冲区的文件提交上去的地方

— 查看状态

git status

这个命令会告诉我们哪些被修改过的文件被添加到了缓冲去，哪些被修改的文件没有。

On branch master
Your branch is behind 'origin/master' by 1 commit, and can be fast-forwarded.
  (use "git pull" to update your local branch)

Untracked files
  (use "git add <file>..." to include in what will be committed)

        blog.html
        blog/hugo.html
        blog/hugo/
        blog/learninggo.html
        blog/page/1.html
        categories.html

nothing added to commit but untracked files present (use "git add" to track)

Untracked files ： 这个表示这些文件还未被添加过，这是用git diff反应不出来差别的。

Modified : 原本文件就存在，只是被修改了。

这个时候可以把没有添加进缓冲区的文件添加进去。

— 查看修改了什么

git diff <file>

一般来说我们都会看看自己一天下来对代码做了什么，这个命令就可以查看差异

一般提交前要：

  1. 查看差异

  2. 然后再添加到缓冲区

  3. 然后查看一下状态

  4. 然后提交

— 查看历史记录

我们在历史记录里面可以查看每一次提交

在Github上就可以很方便的看代码最近由谁更新了，具体更新了什么(这也就是为什么要在commit上写一些提示信息，不然别人不知道这个提交的作用是什么)

还有commit的ID，这个可以用来版本回退。

git log

举个例子：

commit c7ae5e5cdec219f28ff3a4ac9201436739c16610
Author s
Date   Thu Mar 14 113543 2019 +0800

    Create README.md

commit dd377e1e1782a3a8bf027aedb4160040e3fa932e
Author W
Date   Thu Mar 14 113258 2019 +0800

    first commit

— 版本回退

git reset

有两种方式

  1. 用HEAD，HEAD表示版本(HEAD是指向master)，HEAD^上一版本，HEAD^^上两版本，以此类推，HEAD～100就是前100的版本

    git reset --hard HEAD^

  2. 用commit的ID号，commit的ID号不用全部填完，但也不要填太少

    git reset --hard dd377e

如果我们回退后又后悔了，可以用git reflog，这个命令记录了我们用的每一次命令,从这里可以看到之前被抛弃的版本commitID

— 撤销修改

git checkout -- <file>

这里有两种情况：

  1. 一种是readme.txt自修改后还没有被放到暂存区，现在，撤销修改就回到和版本库一模一样的状态；

  2. 一种是readme.txt已经添加到暂存区后，又作了修改，现在，撤销修改就回到添加到暂存区后的状态。

总之，就是让这个文件回到最近一次git commit或git add时的状态。

这个命令本质上就是把版本库的对应文件覆盖到工作区

还有一个可以将添加到缓冲区的文件扯下来

git checkout HEAD -- <file>

—删除文件

git rm -- <file>

git commit -m <message>

远程库(Github)

— 添加远程库

git remote add origin git@github.comusername/repo-name.git

注意要将 - username 换成Github的用户名 - reponame 换成在github上的仓库名

远程库的名字就是origin，这是Git默认的叫法

— 推到远程库

git push

把本地库的内容推送到远程，用git push命令，实际上是把当前分支master推送到远程

由于远程库是空的，我们第一次推送master分支时，加上了-u参数，Git不但会把本地的master分支内容推送到远程新的master分支，还会把本地的master分支和远程的master分支关联起来，在以后的推送或者拉取时就可以简化命令。

当我们本地commit后，就可以用这个命令上传到服务器(远程仓库)

git push origin master

— 克隆一个本地库

git clone git@github.comusername/repo-name.git

这种使用ssh协议

或者

git clone https//github.com/username/repo-name.git

分支管理

分支管理可以让我们自己还未完成的工作先不合并到master，自己在master之外创建一个分支，把代码提交到创建分支。等到一个功能的代码完全实现了的时候，就可以和master合并了。

而创建分支，合并分支，删除分支原理上都是对指针的操作，所以速度非常快。

— 创建分支

git branch <new branch>

或者 创建并且切换

git checkout -b <new branch>

— 切换分支

git checkout <branch>

— 查看当前分支

git branch

— 合并分支

git merge <branch>

如果是要在master合并别的分支，就要保证现在处于master分支上

— 删除分支

git branch -d <branch>

这个是用来删除被合并过的分支

如果是没被合并过的分支，需要

git branch -D <branch>

— Git鼓励使用分支

因为Git在这方面操作是很快的，所以我们可以为了保证master分支的安全，可以创建一个分支，等代码稳定后再合并到master

分支冲突

— 如何产生分支冲突

分支冲突一般就是我们在其它分支改了某个内容并提交，又在master分支改了相同的内容并且提交，这个时候就会发生冲突。

而冲突只是一种警告，告诉我们在不同的分支改了相同的地方(其实它是在问我们究竟要遵循谁的改动)，这个时候我们在master打开那个冲突文件，将我们想要的最终内容填上去，再次提交就可以了。

所以冲突是为了提醒我们。

在我们解决冲突的时候，vscode会在代码冲突的地方给出选项 current和incoming

  • Current 是指版本库的
  • incoming 是指工作区的

— 查看分支合并图

git log --graph

使用

git log --graph --pretty=oneline --abbrev-commit

结果

*   34a0de2 (HEAD -> master) comflict fixed
|\
| * 73341eb (dev) commit on the dev
* | 6a2092b commit on the master
|/
* 958a051 Create README.md

分支策略

我们在上面讲到的合并方法用的都是Fast-Forward

但这种模式下，删除分支后会丢掉分支信息。

如果要强制禁用Fast forward模式，Git就会在merge时生成一个新的commit，这样，从分支历史上就可以看出分支信息。

git merge --no-ff -m "merge with no-ff" dev

直接用Fast forword模式，再用git log --graph --pretty=oneline --abbrev-commit去查看得到

* 640191b (HEAD -> master) dev commit
* 958a051 Create README.md

用--no-ff方法得到

*   5a6b0f5 (HEAD -> master) master commit
|\
| * 2644587 (dev) dev commit
|/
* 640191b dev commit
* 958a051 Create README.md

在实际开发中，我们应该按照几个基本原则进行分支管理：

首先，master分支应该是非常稳定的，也就是仅用来发布新版本，平时不能在上面干活；

那在哪干活呢？干活都在dev分支上，也就是说，dev分支是不稳定的，到某个时候，比如1.0版本发布时，再把dev分支合并到master上，在master分支发布1.0版本；

你和你的小伙伴们每个人都在dev分支上干活，每个人都有自己的分支，时不时地往dev分支上合并就可以了。

Bug分支

我们在某一个分支上工作，突然有个紧急任务需要完成，这个时候手头的工作还未完成，就要把现场存储起来

git stash

然后我们再切换分支，解决问题，回到原来的分支

git stash list

这个会显示有多少工作现场被存储了。

把工作线程弹出来

git stash pop

git stash apply stash@{0} //看git stash list选择想要恢复的

多人协作

查看远程库信息

git remote

or

git remote -v

推送分支

git push origin master

推送分支不一定要推master

git push origin dev

抓取分支

一般来说，我们clone下来的只有master分支

git clone git@github.com:username/repo-name.git

如果我们需要在另一个分支上进行开发(前面说过，master一般都是稳定后才上传的，开发一般会有别的分支)

所以我们要拉取另一个分支

git checkout -b dev origin/dev

这样我们就可以在dev上修改，并且push到远程

推送冲突

git push origin dev
To github.com:michaelliao/learngit.git
 ! [rejected]        dev -> dev (non-fast-forward)
error: failed to push some refs to 'git@github.com:username/repo-name.git'
hint: Updates were rejected because the tip of your current branch is behind
hint: its remote counterpart. Integrate the remote changes (e.g.
hint: 'git pull ...') before pushing again.
hint: See the 'Note about fast-forwards' in 'git push --help' for details.

代表别人最近的推送和你想推送的东西起了冲突，需要先git pull把最新的提交从origin/dev抓下来，然后在本地合并，解决冲突，在继续推送。

我们在git pull的时候可能会出现这样的错误

There is no tracking information for the current branch.
Please specify which branch you want to merge with.
See git-pull(1) for details.

    git pull <remote> <branch>

If you wish to set tracking information for this branch you can do so with:

    git branch --set-upstream-to=origin/<branch> dev

这句话的意思是，你的分支和github上的github没有建立链接关系。

git branch --set-upstream-to=origin/dev dev

这样就建立了链接，再git pull

拉取下来后就是要解决冲突了，解决了冲突后就可以提交上去了。

因此多人合作的工作模式一般是：

  1. 尝试推送自己的分支

  2. 如果失败，就说明代码库被更新过，拉取下来

  3. 合并并且解决冲突

  4. 推送上去

标签管理

其实就是版本号，为了好让人识别

— 创建标签

git tag <tagname>

用于新建一个标签，默认为HEAD，也可以指定一个commit id

git tag -a <tagname> -m "blablabla..."

用来指定标签信息

git tag

查看所有标签

删除标签

git tag -d v1.0

这种情况是还未上传到远程

我们所加的标签都是在本地的，如果要推到github上

git push origin v1.0

或者一次性把所有标签推出去

git push origin --tags

如果标志上传到了远程，要进行删除就要先从本地删除，然后从远程删除

git tag -d v1.0

git push origin :refs/tags/v1.0

总之

    命令git push origin <tagname>可以推送一个本地标签；

    命令git push origin --tags可以推送全部未推送过的本地标签；

    命令git tag -d <tagname>可以删除一个本地标签；

    命令git push origin :refs/tags/<tagname>可以删除一个远程标签。

使用Github

如果想要在别的优秀开源项目上做自己的DIY

  1. 在别人Github仓库上Fork一个到自己的仓库

  2. 在本地用git clone拉取到电脑上

  3. 改完后就提交到自己的代码库，如果希望项目方能接收这个修改，这个时候就可以发起一个Pull Request
