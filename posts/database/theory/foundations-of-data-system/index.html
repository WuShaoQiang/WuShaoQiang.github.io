Foundations of Data System

2019年9月27日
| 数据库
| 阅读

我们要知道，不同的数据库系统拥有着不同的特性，因此在不同的应用场景下会根据应用所需的特性来选择数据库系统，这也是开发者对数据库必须要了解的地方

这一blog主要是覆盖三个方面

  • Reliability

  • Scalability

  • Maintainability

Reliability

一般来说，我们希望我们的软件

  • 正确执行我们所期望的

  • 就算user操作失误，也能一定程度上容忍

  • 性能在合适的压力下是好的

  • 系统可以防止外界未授权侵入

下面我想先解释一下两个经常搞混的名词

  • Fault，这个词是用来描述整个系统中的某一个部件(模块)出现了问题。当然，如果系统就只是一个模块，那么也就等价于Failure了，但是当今众多系统中，基本都由多个模块组成。

  • Failure，这个词是用来描述整个系统出了问题，说明系统不可用了

所以我们要打造的是一个Fault-tolerance的系统，以防止单个模块出问题影响整个系统

用具体的例子说模块出问题就是，将某一个进程杀掉，看看整个系统有没有因此而不工作了(设想中是要工作的)

Hardware Fault

硬件出错，其中包括

  • 内存满了

  • 断电

  • 磁盘损坏

…

只要是电子原件，都会有坏的可能

所以，如果软件对可用性要求非常苛刻，那么将服务部署在多台机器上是必不可少的，这也是常说的避免单点故障。而且，多台机器部署，在对服务进行升级的时候，可以控制在对外无感知的情况下升级。也就是帮天上的飞机换轮胎

但是，硬件错误，一般只会是单个出错，同时出错的概率是非常低的(自然灾害除外)，所以一般来说，系统应该能够容忍少部分机器进入宕机的状态

Software Fault

是人写的软件，都会有Bug。还记得一句话

你以为Hello World就不会有Bug了吗，等什么时候链接器，编译器出现Bug了，就连Hello World这样的程序也跑不过

所以在设计一个系统的时候，绝对不能假设某一个模块是没问题的，哪怕它的代码再少

那我们应该怎么做可以最大程度减少Failure的可能性呢：

  • 设计这个系统的时候需要有容忍性的设计

  • 测试，必不可少的

  • 隔离，也就是一个模块的Fault不应该引起其它模块的Fault

  • 容忍宕机重启操作

  • 有观测整个系统的Monitor(可能是一些Web，或者命令行)

…等等

Human Error

在设计系统的时候，也不能假设用户就会操作得当，设计这个系统的人也不会是完美，但是可以参考以下几点

  • 定位出错频率最高的地方，并能够重现

  • 在生产环境中也加入测试，但注意是不能影响使用的

  • 错误应该容易被找出

How Important Is Reliability

在一些用户基数比较大的应用上，可靠性可以说是重中之重，服务挂掉意味着流量流失，损失可大可小

但是，设计一个可靠性很强的系统是非常花费人力，物力的。在小公司或者小场景下，还是要平衡两者

可以适当减少一些复杂的可靠性，以提高开发速度

Scalability

如何描述一个系统具有扩展性呢，我们可以简单认为是系统可以通过加机器的方式提升负载能力

当一个系统的负载超出了可承受能力，扩展性就显得尤为重要了

要知道，纵向扩展是越来越昂贵的，而横向扩展的开支是相对可接受的

有一种有趣的系统设计是弹性的，当复杂多的时候，给这个系统的资源也会相应的增多，否则反之。但是实现这样的机制可能会引发其它奇怪的问题

要注意的是，没有任何一种可扩展性设计能够通用，我们必须根据应用的需要来做相应设计，比如我们对产品会预期它的读会比写数据库要多得多，那我们在设计这个数据库系统的时候就可以针对读来做更多的优化和设计

所以在最开始的时候就要有一定的假设去做早期的设计

Maintainability

我们知道，初期开发过程其实相对整个产品周期是非常小的，我们建立了一个应用，可能会根据用户的反馈，或者日常的更新，或者bug修复，一次又一次的维护这个产品，所以维护性也是要在设计和编写代码的时候谨记的一点，不然，到时候就悲剧了

这里对保持可维护性提出了三点参考：

  • Operability： 操作一定要简单，如果设计了对维护团队及其不友好的交互方式，那么迟早会出问题的

  • Simplicity： 还是简单，但是这个简单是指整一个系统能够让工程师容易理解，将一些复杂的东西去除，或者抽象化。这样也能够让新人尽快融入整个系统的开发

  • Evolvability： 可持续发展，如何让代码保持一个健康，可持续发展的状态也是程序员重点考虑的，这里主要涉及到软件设计层面，例如：重构，设计模式等

所以，Maintainability是对整一个开发团队以及相应的测试团队都有影响的

Summary

我们清晰一个点就是，不是所有系统都需要以上三个特性，我们需要明确应用所需特性来进行选择。

切记：没有一个通用的设计能够应付所有的应用场景
