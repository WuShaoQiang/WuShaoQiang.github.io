Linux File And Dir

2019年11月16日
| Linux
| 阅读

一些规则

UNIX的一些文件函数中，经常会出现名字差不多，但是参数不同的函数，它们的作用一般遵从这样的规则

比如

int stat(const char *restrict pathname, struct stat *restrict buf)

int fstatat(int fd, const char *restrict pathname, struct stat *restrict buf, int flag)

在以下情况下，这两个函数的作用相同

  1. fd = AT_FDCWD && pathname是一个相对路径

  2. pathname是一个绝对路径

还可以根据flag的值区分是stat还是lstat，主要是这个标志AT_SYMLINK_NOFOLLOW，当这个标志位被置位的时候，就是lstat，如果没置位，就是stat

以上并不是要求对函数的掌握，而是一种规律，UNIX许多文件操作的函数都会有相类似的结构

Stat结构体

struct stat {
	unsigned long	st_dev;		/* Device.  */
	unsigned long	st_ino;		/* File serial number.  */
	unsigned int	st_mode;	/* File mode.  */
	unsigned int	st_nlink;	/* Link count.  */
	unsigned int	st_uid;		/* User ID of the file's owner.  */
	unsigned int	st_gid;		/* Group ID of the file's group. */
	unsigned long	st_rdev;	/* Device number, if device.  */
	long		st_size;	/* Size of file, in bytes.  */
	int		st_blksize;	/* Optimal block size for I/O.  */
	long		st_blocks;	/* Number 512-byte blocks allocated. */
	long		st_atime;	/* Time of last access.  */
	unsigned long	st_atime_nsec;
	long		st_mtime;	/* Time of last modification.  */
	unsigned long	st_mtime_nsec;
	long		st_ctime;	/* Time of last status change.  */
	unsigned long	st_ctime_nsec;
};

这里稍微删除了一些字段，但保留了主要的字段，这里按顺序一个个说

  1. st_dev: device number(file system)

  2. st_ino: i-node number

  3. st_mode: file type & permission

  4. st_nlink: number of links

  5. st_rdev: device number for special file

  6. st_blksize: best I/O block size

  7. st_blocks: number of disk block size

…

有一些比较简单的字段就不说了

这个结构体是用来描述一个文件的

文件类型

普通文件和目录文件以及包含了我们平时用到的大多数文件，还有一些特殊的文件

  • block special file

  • character special file

  • FIFO

  • socket

  • symbolic link

其中可能FIFO和symbolic link还算在这几个里面比较常用，前者一般用在进程之间的通信，后者是一个链接

st_mode

我们来看一下st_mode关于权限的情况

S_ISUID /* 设置有效UID */
S_ISGID /* 设置组ID */
S_ISVTX /* 限制在目录中删除和重命名文件 */

S_IRUSR
S_IWUSR
S_IXUSR

S_IRGRP
S_IWGRP
S_IXGRP

S_IROTH
S_IWOTH
S_IXOTH

这里总共有四个分组，其中后面三个分组是我们平时看到的，对user、group和other的读写执行权限控制

第一组的S_ISUID和S_ISGID是关于set-user-ID和set-group-ID的提权，一般应用于进程需要切换权限的场景

S_ISVTX应用在/tmp目录下，它实现了每个用户都能在tmp目录下创建和删除文件，但是只能删除自己的文件

文件的RWX代表着

  • R: 读取文件内容

  • W: 写入文件内容

  • X: 执行文件

目录的RWX代表着

  • R: 读取目录的文件名

  • W: 修改目录里的文件(包括创建、删除、重命名)

  • X: 进入目录

其实目录的RWX会比较难理解一点，我们后面再说

新文件和目录的所有权

文件的UID是当前进程的有效ID(eUID)

POSIX.1对组ID提供了两个选择

  1. 进程的有效组ID

  2. 所在目录的组ID(跟随目录)

access

和平时的open或者RW(read-write)操作不同，access使用的是real-user和read-group，而open等使用的是effective-user和effective-group

在一般情况下，real和effective是相同的，但也有不同的时候，比如需要权限切换的时候，effective-user可能是root，而real-user可能是一个普通的用户

access这个函数使用的是real-user的权限

umask

umask是一个掩码，在创建文件的时候，默认会其进行操作，比如

umask = 002(default)

然后我创建了一个文件的权限是777，那么得到的结果将是775，其实就是将other的write权限给剥夺了

chmod

调用这个命令的时候，euid必须要和当前文件的owner一样，或者以root权限调用
