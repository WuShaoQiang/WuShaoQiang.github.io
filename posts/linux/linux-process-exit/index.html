Linux Process Exit

2019年11月4日
| Linux
| 阅读

之前几篇blog对进程的创建(fork&clone)，以及执行(exec)的大概设计和具体的思路，但还没有将进程的退出是怎么回事

进程终止的条件

在大多时候，我们所看到的程序退出包括：

  • 执行完退出

  • 异常退出

但是在Linux下，进程有5种正常终止和3种异常终止

这里的种类虽然多，但是常用的也就三种吧

  1. 调用exit

  2. 自己abort

  3. 内核产生信号

隐式exit

首先，调用exit并不是意味着你每次都要显式的去调用这个函数，其实在很多编程语言里(比如我用的Go)，在main函数里，我通常不会在最后去显式地调用exit。比如HelloWorld

func main() {
    fmt.Println("Hello World")
}

这里应该是编译器帮我在main函数执行完后调用了exit，并且exit status是0

echo $? #用这个验证

这里我也只是猜测，我还没具体考究到底是不是编译器帮我们做了这件事情

但是能肯定的是，我们的程序最后肯定调用了Linux给到的exit

显式exit

这个就很明显了

func main() {
    fmt.Println("Hello World")
    os.Exit(1)
}

通过程序显式的调用exit，同理echo $?可以得到相应的退出码

无论是显式调用还是隐式调用，这种退出方式都是进程的正常退出方式

不正常退出

其实2和3也算是一类，它们都可归类为信号退出

内核产生信号使得进程不正常退出一般是因为进程有非常严重的违法操作

  • 引用了地址空间之外的存储单元

  • 除以0

…

出现了这类操作，内核会产生一个让进程abort的信号

但是Golang将这类不正常退出转换为了正常退出(panic)

所以在Golang里，不正常退出一般都是由外界信号导致的，比如SIGKILL

无论如何

无论进程如何退出，最后都会执行内核中同一段代码

无论是正常还是非正常，进程都会能够让父进程取到自己的退出状态(exit status)

父进程会通过wait来获取子进程的退出状态，但是正常和非正常退出的获取方式有些许不同

POSIX.1规定，用4个互斥的宏来取得进程终止的原因

WIFEXITED(status)
WIFSIGNALED(status)
WIFSTOPPED(status)
WIFCONTINUED(status)

如果一个进程是被信号杀死的(非正常终止)，那么用EXITED来获取status是不可以的，要用SIGNALED，一般来说，都会有相应的API提供使用

但是我们要知道的是，进程的退出没有那么简单，不是所有退出都叫EXITED，我们从上面也可以看见有4种不同的”退出”情况

结语

其实这篇blog主要是为了让读者知道所谓的”退出”也分好几种，如果用不匹配的方式去获取终止状态码可能会出问题。
