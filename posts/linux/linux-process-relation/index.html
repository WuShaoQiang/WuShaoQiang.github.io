Linux Process Relation

2019年11月17日
| Linux
| 阅读

Process Group

Process Group的声明周期是：从Group被创建，到这个Group里面最后一个Process退出这个Group(Terminated或者加入到别的Group中去)

int setpgid(pid_t pid, pid_t pgid)

参数的含义

如果pid和pgid相同，那么pid这个进程就是Process Group Leader

如果pid是0，那么是指当前进程

如果pgid是0，那么pgid = pid

setpgid只能对自己使用，或者是对子进程使用

一般来说，这个函数会在fork之后调用，因为当一个进程调用exec之后，pgid就无法改变了。由于父进程和子进程无法保证执行顺序，因此通常会在两个进程里面都调用setpgid，以确保后面依赖于这个进程组关系的程序不会发生RACE。

关于如何区分是父子进程，根据fork返回值来确认(在不发生错误的情况下)

  • 子进程的fork返回的是0

  • 父进程的fork返回的是子进程的pid

Session

当我们创建一个session的时候，会发生下面三件事情

  1. 进程是Session Leader，并且整个Session中只有它一个进程

  2. 进程变成Process Group Leader，并且pgid = pid

  3. 进程没有controlling terminal

创建一个新的Session的进程必须不是一个Process Group Leader，因此最常见的实现方式是这样的：

  1. 先调用fork，产生子进程，父进程退出

  2. 子进程调用setsid创建新的session

由于子进程不可能是进程组Leader，因此这里可以保证创建新Session的条件成立

Controlling Terminal

一般情况下，我们只需要知道，当我们Login进Linux系统的时候，我们调用命令行的地方(简单来说)就是Controlling Terminal

我们可以运用Process和Session的知识对它大概了解：

  1. Session Leader是建立这个Controlling Terminal的Process

  2. 一个Controlling Terminal只能包含一个foreground process group，能够包含多个background process group

一般来说，我们所按的组合键，都是对foreground process group发信号。我们在shell里面输入(Stdin)是由foreground process group接收的

所以我们在foreground执行的任何前台命令实质上都会产生在session的process group里面

如果一个background process需要读取控制台的输入(Stdin)，也是可以的：

  1. terminal driver会给这个background process发送SIGTTIN信号，这个时候进程就进入Stopped状态

  2. 然后我们再将该进程放置到前台运行fg %n(n是job序号)，这个时候就能够读取到Stdin了。

所以其实还是前台进程组来读取

如果一个background process需要在终端输出(Stdout)，默认的配置是允许的，所以并不用配置。如果要禁止后台进程输出到终端，那么可以使用stty tostop来禁止

同理，如果禁止了后台进程输出Stdout，Terminal driver在检测到后台对Stdout进行输出的时候，会给这个进程发送SIGTTOU信号，然后后台进程就会进入Stopped状态。

同理，需要调到前台才能继续运行(继续运行则是给进程发送SIGCONT信号)

Orphaned Process Groups

只要满足以下，就不算是Orphaned Process Group：

  • 只要组内有一个进程的父进程是来自不同的组，但在一个Session。因为这样有可能某一个父进程会重启这个停止的进程

当进程组要被当做孤儿进程组，POSIX标准要求，新的孤儿进程组要接收一个SIGHUP的信号，然后再接收一个SIGCONT

在孤儿进程组里的进程是无法接收Stdin或者输出到Stdout的，像之前所说，后台进程想要实现这两者，状态会先转换到Stopped，这个进程可能永远不会继续执行下去
