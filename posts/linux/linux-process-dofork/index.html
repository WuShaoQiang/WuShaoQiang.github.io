Linux Process do_fork

2019年10月24日
| Linux
| 阅读

上一篇说fork、clone的时候，可以看到它们都是调用了do_fork，这也很好的解释了为什么这两个系统调用如此相似了，也解释了很多我们平时听过的结论(不再赘述)

do_fork大致流程

do_fork的大致代码如下

/*
 *  Ok, this is the main fork-routine.
 *
 * It copies the process, and if successful kick-starts
 * it and waits for it to finish using the VM if required.
 */
long do_fork(unsigned long clone_flags,
	      unsigned long stack_start,
	      struct pt_regs *regs,
	      unsigned long stack_size,
	      int __user *parent_tidptr,
	      int __user *child_tidptr)
{
	struct task_struct *p;
	int trace = 0;
	long nr;

	/*
	 * Do some preliminary argument and permissions checking before we
	 * actually start allocating stuff
	 */
    // 这里是一些clone_flag的检查
	if (clone_flags & CLONE_NEWUSER) {
		if (clone_flags & CLONE_THREAD)
			return -EINVAL;
		/* hopefully this check will go away when userns support is
		 * complete
		 */
		if (!capable(CAP_SYS_ADMIN) || !capable(CAP_SETUID) ||
				!capable(CAP_SETGID))
			return -EPERM;
	}

	...
    // 复制一个进程
	p = copy_process(clone_flags, stack_start, regs, stack_size,
			 child_tidptr, NULL, trace);
	/*
	 * Do this prior waking up the new thread - the thread pointer
	 * might get invalid after that point, if the thread exits quickly.
	 */
	if (!IS_ERR(p)) {
		...
        // 拿local pid
		nr = task_pid_vnr(p);

		...
        // 唤醒进程
		wake_up_new_task(p, clone_flags);

		...

		if (clone_flags & CLONE_VFORK) {
			freezer_do_not_count();
            // 唤醒因为该进程而睡眠的进程
			wait_for_completion(&vfork);
			freezer_count();
			tracehook_report_vfork_done(p, nr);
		}
	} else {
		nr = PTR_ERR(p);
	}
	return nr;
}

  1. 先简单的检查flag是冲突

  2. 复制进程(也会检查flag)

  3. 拿到PID

  4. 唤醒进程实际上是将进程放入可执行队列

  5. 如果有CLONE_VFORK标志，会唤醒其它进程

copy_process

从这个函数的名字可以很好的知道它所做的事情

它内部大概会做这么几件事情

  1. 先检查标志集是否有不符合的

  2. 在调用dup_task_struct

  3. 检查资源限制

  4. 初始化task_struct

  5. 调用sched_fork为了调度器能够有机会对新的进程进行设置

  6. 按照flag来复制资源

关于标志集的冲突这里不会涉及到，这一部分大家可以先去了解完大致的标志再来看源码

dup_task_struct是一个比较重要的函数，它将复制一个子进程，这个子进程只有栈是不同的

检查资源现在的代码也比较简单

if (atomic_read(&p->real_cred->user->processes) >=
			task_rlimit(p, RLIMIT_NPROC)) {
		if (!capable(CAP_SYS_ADMIN) && !capable(CAP_SYS_RESOURCE) &&
		    p->real_cred->user != INIT_USER)
			goto bad_fork_free;
	}

大概的思路就是检查rlimit，如果超过了limit，再去检验其权限(有一些权限可以在超过limit后也能继续执行)

在copy_process里，还初始化一些父子进程不共享的资源，比如：

  • sibling, children

  • cpu_timers

  • 待决信号列表pending

在调用了sched_fork之后，进程的状态被设置为TASK_RUNNING

后面就是一些资源的共享了，有clone标志的资源会被复制，并且count=1，这里的count是为了记录这个资源被引用的次数，以便于在没有引用的时候及时回收

线程与进程的处理细节

下面主要是copy_process里面的一些代码段对线程和进程的处理区别，对CLONE_THREAD置位的flag就是线程的处理

p->pid = pid_nr(pid);
p->tgid = p->pid;
if (clone_flags & CLONE_THREAD)
    p->tgid = current->tgid;

我们可以看到这是对pid和tgid的赋值。tgid一开始赋值是pid(在单线程里是成立的)，然后后面判断这个进程是否是线程，如果是线程，那么它的tgid应该跟父进程(也就是current)的tgid一样(同一线程组)

区别：

  • 进程的tgid就是它的pid(刚创建的进程都是单线程的)

  • 线程的tgid要跟随创建者(调用这个函数的current)

/* CLONE_PARENT re-uses the old parent */
if (clone_flags & (CLONE_PARENT|CLONE_THREAD)) {
    p->real_parent = current->real_parent;
    p->parent_exec_id = current->parent_exec_id;
} else {
    p->real_parent = current;
    p->parent_exec_id = current->self_exec_id;
}

上半部分是线程的处理方式，下半部分是进程的处理方式

区别：

  • 线程的parent应该跟随进程的调用者的parent，因为在线程的概念下，它们属于”同一进程”，所以它们的父进程应该都一样

  • 进程的创建，父进程应该是调用者current

if (clone_flags & CLONE_THREAD) {
    ...
    p->group_leader = current->group_leader;
    list_add_tail_rcu(&p->thread_group, &p->group_leader->thread_group);
}

这里讲述的是，线程的组长和调用者的组长是一样的，这个和第一个所说的有点类似

区别：

  • 线程之间共享一个组长(在概念上，线程组是在一个进程)

  • 进程的组长是进程本身(在没调用setpgid前)
