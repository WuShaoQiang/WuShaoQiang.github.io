Linux Process Identifier

2019年10月23日
| Linux
| 阅读

接触Linux系统也有一年时间了，大部分接触过Linux系统的人也都清除Linux依靠各种ID来实现区分

这里面包括

  • PID

  • SID

  • PGID(Process Group ID)，值为进程组组长的PID

  • TGID(Thread Group ID)，如果是单线程，则与PID一样

初学者也能知道，PID是进程的唯一标识(Identifier)，再结合之前说的Namespace，我们应该换一种说法PID是某个Namespace下的进程唯一标识

引入了Namespace之后，我们对ID会有两种

  • Global ID

  • Local ID

其实它们也类似，只是Global ID是boot的时候的第一个Namespace，因此我们叫它做整个系统的Namespace，Global也因此得名

Local ID就是不同的Namespace下面的ID，也非常好理解

task struct

在task struct里面，有

1278     pid_t pid;                     
1279     pid_t tgid;

其中pid_t一般是32bit的int，因此可以包含足够的ID

但这里面没有pgid以及sid，这些是存储在了signal字段里面

PID不仅仅是数字

在我们大部分人的认知下，PID就是一个整数，用来标识进程

但是在Linux下，需要维护PID之间的关系，比如所处的Namespace

Linux下的PID Namespace

在Linux下的PID Namespace是有层级关系的，init所处的global PID Namespace是处于第0级，要创建新的PID Namespace就会在其父进程的Namespace的等级里加一

也就是说Namespace level低的能够看到Namespace level高的，反过来则不行。这带来的结果就是，一个进程(Namespace.level > 0)会在不同层级的Namespace上有该层级的唯一PID，所以一个进程有多个PID，分别在不同的Namespace

struct pid_namespace {
	struct kref kref;

	struct task_struct *child_reaper;
	
	unsigned int level;
	struct pid_namespace *parent;
...
};

这里省略了一些字段，简单粗暴的解释一下字段相关意思

  • kref可以简单理解为计数器，记录这个Namespace被引用了多少次

  • child_reaper是一个task struct，是这个Namespace的”init”进程

  • level表示Namespace所处的层级

  • parent就是它的父Namespace，因为我们知道，Namespace是随着进程的创建来的，也就是指向调用了fork或者clone的那个进程的Namespace

Linux下的PID

struct upid {
	/* Try to keep pid_chain in the same cacheline as nr for find_vpid */
	int nr;
	struct pid_namespace *ns;
	struct hlist_node pid_chain;
};

struct pid
{
	atomic_t count;
	unsigned int level;
	/* lists of tasks that use this pid */
	struct hlist_head tasks[PIDTYPE_MAX];
	struct rcu_head rcu;
	struct upid numbers[1];
};

先看一下相对简单的upid，这里面的字段分别代表

  • nr是我们看到的进程PID

  • ns就是相应的PID Namespace

  • pid_chain是一个双向链表(暂时不清楚是什么用处)

再看pid，字段：

  • count是一个引用计数器(为什么不用kerf？还没搞清楚kerf的实现)

  • tasks是一个hlist_head的数组，其中hlist_head其实就是包含一个hlist_node作为它的first字段，所以就是多个双向链表的开头。因为一个ID可能用于不同的Type。这里主要是为了当拿到数字相同但是类型不同的task struct，比如我要PID为12345的进程和要PGID是12345的进程显然要不同处理

struct hlist_head {
	struct hlist_node *first;
};

enum pid_type
{
	PIDTYPE_PID,
	PIDTYPE_PGID,
	PIDTYPE_SID,
	PIDTYPE_MAX
};

  • level和Namespace的level相似，代表着pid所处的Namespace level

  • numbers是一个upid的数组，因为这个数组是结构体的最后一个元素，可以随时再后面增加。每一个元素对应着一个Namespace，这里也就验证了一个进程对应多个PID

这里有一个问题还不是很清晰的被解决。hlist_node的作用是什么，如何发挥作用

解: 按照书本上的说法，这里存储的是task struct，这也其实就说得过去了

task struct下的PID存储在哪

struct pid_link pids[PIDTYPE_MAX]

其中pid_link是

struct pid_link
{
    struct hlist_node node;
    struct pid *pid
};

pid会指向一个刚才说的pid结构体

所以一个task_struct可以很轻松的拿到pid的结构体

小结

这里主要是要认识到Linux内核里面是如何表达pid这个概念的，以及它和进程的关系是如何的

从Linux的使用角度来说，我们可以简单理解为

  • 通过进程，我们要得知它的PID

  • 通过PID，我们要得知是哪个进程(在当前Namespace下)

后面我会详细记录一下Linux内核实现上面两个功能的大概思路

Process -> PID

通过task_struct查找PID是相对简单的

因为task_struct字段中就包含了pid_link->pid

再通过对比pid和对应Namespace的level就可以通过pid.numbers[level]来得到相应的upid，然后再比较upid里的Namespace是否和我们传入的Namespace相同，相同则返回upid.nr

这里只是大概的思路，其实内核里面还有很多函数，可以在pid.c文件上查看

PID Number-> Process

通过一个PID整数来找到对应的进程，大概是下面的步骤

  1. 首先会有一个pid_hash，这里会通过PID和Namespace来算出一个索引，然后最终会找到pid的结构体

  2. 找到pid的结构体后，我们通过它的tasks字段，之前有说过这个字段是存储task_struct的因此我们可以找到对应的进程
